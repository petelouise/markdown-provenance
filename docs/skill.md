# Markdown Provenance (MDP) — Agent Guide

This document describes the MDP syntax for marking text provenance in Markdown notes. When you create or edit a Markdown note in a vault where this convention is in use, you should apply these markers to any text you generate or modify.

---

## What it is

MDP lets every piece of text in a note carry a label indicating who produced it: an AI assistant, the human author (user), an external source, or unknown. The plugin renders these labels as subtle background tints in Obsidian's editor and reading view. The raw markers are valid Markdown — they degrade gracefully as literal text in any other viewer.

---

## The four provenance types

| Sigil | Name | Meaning |
|---|---|---|
| `%a` | assistant | Written by an AI assistant (you) |
| `%u` | user | Written by the human author of the note |
| `%q` | external | Reproduced from a third-party source |
| `%?` | unknown | Provenance genuinely unclear |

The boundary rule: **who produced the text**, not its relationship to a source.
- If the human paraphrases someone else → `%u`
- If you (the AI) paraphrase someone else → `%a`
- If text is reproduced verbatim or near-verbatim from a third party → `%q`

**Backward-compat note:** `%s` is accepted as an alias for `%u` (user). New content should use `%u`.

---

## Syntax

### Inline spans

```
%a{Text generated by an AI assistant.}
%u{Text written by the human author.}
%q{Text from an external source.}
%?{Text of unclear origin.}
```

Spans can be nested. Inner provenance overrides outer:

```
%a{This is AI text, but %u{this correction is human}, back to AI.}
```

### Block markers

When an entire paragraph has a single provenance, prefix every line with `%X> ` (like Markdown's `>` blockquote syntax). The block continues as long as consecutive lines carry the same prefix; a blank line or a line without the prefix ends it.

```
%a> This entire paragraph is AI-generated.

%a> This one too.
%a> It spans multiple lines; every line carries the prefix.

%u> This paragraph is human-authored.
```

For multi-paragraph regions under the same provenance, use a fenced form:

```
%%%a
This is AI paragraph one.

This is AI paragraph two. Still inside the fence.
%%%
```

The closing `%%%` (alone on a line) ends the fence regardless of the opening sigil. Inline spans inside a block or fence still work and override the block provenance for their span:

```
%a> The assistant wrote this. %u{I inserted this correction.} The assistant continued.
```

**When to use blocks vs. inline spans:**
- Use inline `%a{...}` for short runs of text within a paragraph the human is primarily authoring.
- Use `%a>` for paragraphs you (the assistant) wrote entirely.
- Use `%%%a … %%%` for sections of several paragraphs you wrote entirely.

### Escaping

- `\%` → literal `%` (not a span opener)
- `\}` → literal `}` inside a span

### Frontmatter default

A note can declare a default provenance type in its YAML frontmatter. Text that matches the default needs no explicit marker.

```yaml
---
provenance: assistant
---
```

Valid values: `assistant`, `user`, `external`, `unknown`.

Accepted aliases: `ai` → `assistant`, `human` → `user`, `self` → `user`, `quote` → `external`.

**Example:** A note with `provenance: assistant` means all unmarked text is assumed AI-generated. Only human corrections need to be marked: `%u{this part was edited by the author}`.

---

## Rules for agents

When you write or edit a Markdown note:

1. **Mark everything you write.** Use inline `%a{...}` for short runs of text, `%a>` for whole paragraphs you authored, and `%%%a … %%%` for multi-paragraph sections you authored — unless the note's frontmatter sets `provenance: assistant`, in which case you may leave your text unmarked.

2. **Do not re-mark text that was already there.** If user-authored text (`%u{...}` or unmarked in a `provenance: user` note) exists, leave it as-is unless you are explicitly rewriting it.

3. **When you rewrite human text,** wrap your replacement in `%a{...}` so the author can see what changed.

4. **When you quote a source verbatim or near-verbatim,** use `%q{...}` regardless of who instructed you to include it.

5. **Preserve existing markers.** Never strip or flatten provenance markup that is already present.

6. **If in doubt about a passage's origin,** use `%?{...}`.

7. **Choose the right granularity.** A full paragraph you wrote → `%a>`. A sentence you inserted in someone else's paragraph → `%a{...}`. A multi-paragraph section → `%%%a … %%%`.

## Examples

### AI-drafted note with human edits

```yaml
---
provenance: assistant
---

This paragraph was generated by the AI and needs no marker.

%u{This sentence was added by the human author.}

The AI continued here, still unmarked.
```

### Human note with AI assistance

```yaml
---
provenance: user
---

This is the author's own writing.

%a{This paragraph was suggested by the AI assistant.}

Back to the author's voice. %a{One more AI-generated sentence here.}
```

### Mixed with an external source

```
The philosopher wrote: %q{The unexamined life is not worth living.}

%a{This interpretation was generated by the AI.}
```
